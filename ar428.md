# App Store Review Critique — StatScout (Build 22)

> Review Date: April 28, 2026  
> Build: 1.0 (22) — commit `HEAD`  
> Reviewer: App Store Review Critic (AR-428)  
> Scope: iOS app binary, backend data pipeline, CI/CD integrity, marketing web presence, and cross-layer coherence.  

---

## Executive Verdict: Conditional Pass with Notes

StatScout is a lean, functional iOS 17+ SwiftUI app that visualizes Baseball Savant percentile rankings from a Supabase-backed nightly feed. The core user journey — launch → browse leaderboard → filter by category → tap player → view percentile bars and standard stats — is coherent and largely crash-free.  

**Why not an unconditional pass:** The app still ships with a non-interactive UI element that resembles a tappable button, a hardcoded backend endpoint in the compiled binary, thin accessibility coverage for its primary visual content (percentile bars), and marketing copy on the public website that overstates data freshness ("Live Leaderboards"). These are the exact friction points that trigger App Store Review rejections under Guideline 4.0 (Design) and Guideline 2.3 (Accurate Metadata).  

Most of the critical bugs from prior audits have been fixed. What remains is a layer of polish, configuration hygiene, and accessibility completeness that separates a TestFlight build from a store-ready product.

## Guideline 2.1 — Performance: App Completeness

### Issue 2.1-1: Ghost interactive element on every player profile

- **Location:** `StatScout/Views/PlayerProfileView.swift:95`  
- **Observation:** The "ⓘ" glyph (circled lowercase i) appears immediately to the right of the season label inside the "PERCENTILE RANKINGS" section header. It uses `SavantPalette.linkBlue`, a hyperlink color, and is rendered at `11pt` with no tappable area.  
- **Expected behavior:** Either make it a tappable `Button` that presents a sheet explaining how Baseball Savant percentiles are calculated, or remove the glyph entirely. A colored circled-i that does nothing when tapped is a classic 4.0 rejection trigger — it trains the user to expect an info panel that never arrives.  
- **Severity:** Medium (rejection risk)  

### Issue 2.1-2: No data-freshness signal on primary screens

- **Location:** `StatScout/Views/DashboardView.swift`, `StatScout/Views/TeamView.swift`  
- **Observation:** The only place a user can discover when data was last refreshed is two taps deep in the About tab (`AboutView.swift:67`). The dashboard header, leaderboard, and team rosters show no relative-time label (e.g., "Updated 6h ago").  
- **Expected behavior:** Add a lightweight freshness pill or subtitle to the `DashboardView` header (e.g., beneath the "LEADERBOARD" section bar) showing `viewModel.freshnessText`. The view model already computes this string; the UI simply never renders it outside the About tab.  
- **Severity:** Medium  

### Issue 2.1-3: Metric bars are invisible to assistive technology

- **Location:** `StatScout/Views/Components.swift:291-310` (`PercentileBarMini`), `StatScout/Views/Components.swift:69-110` (`MetricBar`)  
- **Observation:** `PercentileBarMini`, used in every leaderboard row, is a pure `GeometryReader` + `RoundedRectangle` composition. It carries no `.accessibilityLabel`, `.accessibilityValue`, or `.accessibilityHidden(true)` modifier. A VoiceOver user hears the player name and team, then silence where the percentile bar lives. `MetricBar` at least shows the numeric percentile in an adjacent `Text`, but the bar itself is unlabeled.  
- **Expected behavior:** Attach `.accessibilityLabel("92nd percentile")` and `.accessibilityHidden(false)` to every bar view, or hide the decorative bar from VoiceOver and ensure the adjacent text is read. For `LeaderboardTableRow`, the `PercentileBarMini` must expose its value because there is no separate percentile text node.  
- **Severity:** Medium (Guideline 4.0 accessibility)  

### Issue 2.1-4: Team roster rank lacks league context

- **Location:** `StatScout/Views/TeamView.swift:24`  
- **Observation:** Inside a team roster, every player is ranked `#1`, `#2`, `#3` relative to that team only. A user sees "#1" for the best Royal and "#1" for the best Dodger, but the numbers are not comparable. There is no global percentile or league rank displayed.  
- **Expected behavior:** Add a secondary column or inline badge showing the player's `overallPercentile` (e.g., "#1 · 94th PCTL") so the user understands team-relative vs league-relative standing.  
- **Severity:** Low  

### Issue 2.1-5: Empty standard-stats message can mislead

- **Location:** `StatScout/Views/PlayerProfileView.swift:139-144`  
- **Observation:** When a pitcher has no standard stats (because FanGraphs ingestion failed or the player does not qualify), the empty-state card reads: "Traditional stats are not available for this player in the current data feed." For many pitchers who will never meet the `qual=100` PA threshold for hitters, this message is accurate but sounds temporary.  
- **Expected behavior:** Differentiate between "temporarily unavailable" (data pipeline delay) and "not applicable" (player type mismatch). At minimum, reword to "Traditional stats unavailable for this player."  
- **Severity:** Low

## Guideline 2.3 — Accurate Metadata

### Issue 2.3-1: Marketing site claims "Live Leaderboards"

- **Location:** `docs/index.html:133`  
- **Observation:** The public-facing marketing page includes a feature card titled "Live Leaderboards" with the subtitle "Filter the entire league across Hitting, Pitching, and Running to find top performers." The actual data refreshes nightly via GitHub Actions at 14:00 UTC. There is no websocket, push, or real-time ingestion.  
- **Expected behavior:** Rename to "Nightly Leaderboards" or "Updated Daily" to avoid misleading users who expect sub-minute refresh intervals. App Store Review increasingly cross-references marketing claims against actual app behavior during the review session.  
- **Severity:** Medium (rejection risk under 2.3.7 — Misleading Content)  

### Issue 2.3-2: App Store badge is a dead link

- **Location:** `docs/index.html:154`  
- **Observation:** The "Download on the App Store" SVG badge wraps an `<a href="#">`. Once the app is approved, this must deep-link to the actual App Store product page.  
- **Expected behavior:** Replace `#` with the live App Store URL before launching the marketing site.  
- **Severity:** Low

## Guideline 2.4 — Hardware & Software Compatibility

### Issue 2.4-1: Forced dark mode without adaptive palette

- **Location:** `StatScout/StatScoutApp.swift:30`  
- **Observation:** The app applies `.preferredColorScheme(.dark)` at the `WindowGroup` root. The custom palette (`SavantPalette.canvas`, `.surface`, `.ink`) uses near-white backgrounds and near-black text, which renders correctly under the forced-dark system chrome. However, if Apple ever changes the behavior of `.preferredColorScheme` or the user has Smart Invert enabled, the palette is not truly adaptive.  
- **Expected behavior:** This is acceptable for a brand-specific dark theme, but add a code comment explaining the intentional light-on-light canvas design so future maintainers do not "fix" it to true dark grays.  
- **Severity:** Low (informational)  

### Issue 2.4-2: No iPad or landscape support

- **Location:** `StatScout/Info.plist:44-46`, `project.yml:27`  
- **Observation:** `UISupportedInterfaceOrientations` is locked to portrait, and `TARGETED_DEVICE_FAMILY: "1"` limits to iPhone. The leaderboard and team grid would work fine in landscape or on iPad with minor layout tweaks.  
- **Expected behavior:** Not a rejection, but expanding to iPad and landscape is a quick win for App Store featuring eligibility.  
- **Severity:** Low

## Guideline 4.0 — Design

### Issue 4.0-1: "Worst" column header is semantically misleading for inverse metrics

- **Location:** `StatScout/Views/MetricLeadersView.swift:55`  
- **Observation:** In the Metric Leaders tab, every metric shows a "BEST" and "WORST" player based on percentile descending. For pitching metrics like `BB%` (walk rate), a high percentile actually means the pitcher walks *fewer* batters (good). The "WORST" label in the UI refers to the lowest percentile, which is still correct in percentile terms, but a casual fan may misread it as "this player has the worst walk rate" when the column actually shows the worst *ranking*.  
- **Expected behavior:** Rename the column header to "LOWEST" or add a small tooltip icon that explains the column is sorted by percentile rank, not raw value directionality.  
- **Severity:** Low  

### Issue 4.0-2: Category filter chips lack count badges

- **Location:** `StatScout/Views/Components.swift:137-158` (`CategoryFilter`)  
- **Observation:** The dashboard shows five category chips: All, Hitting, Pitching, Fielding, Running. A user tapping "Fielding" may see only 3–5 players with no warning. The chip layout gives every category equal visual weight.  
- **Expected behavior:** Append a count to each chip label (e.g., "Hitting (247)") or dim chips with zero qualifying players. This prevents the "empty result after one tap" surprise.  
- **Severity:** Low  

### Issue 4.0-3: No metric glossary for casual fans

- **Location:** `StatScout/Views/PlayerProfileView.swift` (Advanced tab)  
- **Observation:** Metrics like "xwOBA", "OAA", "Barrel%", and "Chase%" are displayed as raw abbreviations. There is no in-app definition, tooltip, or link to a glossary. The target audience includes "fans and media," but casual fans will bounce at the first unknown acronym.  
- **Expected behavior:** Add a tappable info accessory to each `MetricBar` row (or a long-press context menu) that presents a one-sentence definition and a link to the Baseball Savant glossary.  
- **Severity:** Medium  

### Issue 4.0-4: No onboarding or empty-state guidance for first launch

- **Location:** `StatScout/StatScoutApp.swift`  
- **Observation:** On first launch the app immediately fires a network request and shows either data or a spinner. There is no 2–3 card onboarding explaining what percentiles are, how often data updates, or that no account is required.  
- **Expected behavior:** Add a conditional `OnboardingView` shown once per install (tracked via `AppStorage`) that explains the app's value proposition and gestures.  
- **Severity:** Low

## Guideline 5.1.1 — Data Collection & Storage (Privacy)

### Issue 5.1.1-1: Privacy manifest may not be bundled as a resource

- **Location:** `StatScout/PrivacyInfo.xcprivacy`, `project.yml:16`  
- **Observation:** The `.xcprivacy` file exists and declares empty `NSPrivacyCollectedDataTypes` and `NSPrivacyAccessedAPITypes` arrays, which is correct for an app that collects nothing and uses no required-reason APIs. However, `project.yml` sources the recursive `StatScout` directory; XcodeGen may treat the `.xcprivacy` file as a source file rather than a bundle resource. If it is not copied into the app bundle, App Store Connect will reject the binary.  
- **Expected behavior:** Explicitly add `StatScout/PrivacyInfo.xcprivacy` to the `resources` array in `project.yml`, or verify in the generated `project.pbxproj` that it appears in a `PBXResourcesBuildPhase`.  
- **Severity:** High (rejection risk)  

### Issue 5.1.1-2: No per-player privacy attribution for third-party images

- **Location:** `StatScout/Models/Player.swift:165` (`headshotURL`)  
- **Observation:** Headshot images are loaded from `midfield.mlbstatic.com` (MLB CDN). The app does not proxy or cache these images locally beyond the default `AsyncImage` ephemeral memory cache. The user's IP address is therefore exposed to MLB Advanced Media on every profile visit.  
- **Expected behavior:** Document this in the privacy policy (already present in `docs/privacy-policy.html`) and add a note to `PrivacyInfo.xcprivacy` under `NSPrivacyCollectedDataTypes` if MLB's CDN is considered a third-party data receiver. Since the policy already discloses that "standard technical information (such as your IP address)" is transmitted to fulfill server requests, the current disclosure is probably sufficient, but the manifest should be updated if Apple ever requires network endpoint disclosure.  
- **Severity:** Low (informational)

## Backend & Data Integrity (Indirect App Store Impact)

### Issue BE-1: Hardcoded Supabase project URL in compiled binary

- **Location:** `project.yml:31`, `project.yml:35`  
- **Observation:** `SUPABASE_URL: https://babzqsbmcunrezsdpyng.supabase.co` is a literal string in both Debug and Release build configurations. This URL is injected into `Info.plist` at build time and read by `StatScoutApp.swift:18`. While RLS prevents writes without the service role key, the endpoint is public, scrapable, and rate-limitable.  
- **Expected behavior:** Change the build setting to `SUPABASE_URL: $(SUPABASE_URL)` and inject the URL via environment variable or CI secret, exactly as is done for `SUPABASE_ANON_KEY`.  
- **Severity:** Medium (security / operational hygiene)  

### Issue BE-2: Some metric raw values may render as empty strings

- **Location:** `backend/ingest.py:169-208` (`raw_stat_value`)  
- **Observation:** The heuristic that distinguishes raw stat values from percentiles relies on decimal-component detection (`f != int(f)`) and out-of-range checks (`f < 0 or f > 100`). For metrics where a raw value is an exact integer (e.g., a 96.0 mph exit velocity that happens to be rounded), the function returns `None`, and the UI sees `value: ""`. The app then hides the value entirely, showing only the label and percentile bar.  
- **Expected behavior:** The `build_metrics` function should log when `raw_stat_value` fails, and the heuristic should favor the raw column if a companion `_value` suffix exists. Over time, consider switching to an explicit column map from pybaseball rather than heuristics.  
- **Severity:** Low (data quality)  

### Issue BE-3: `games` array is always empty in production data

- **Location:** `backend/ingest.py:466`  
- **Observation:** `merge_player_row` hardcodes `"games": []` for every player. The iOS model fully supports `GameTrend` with `latestGame`, `weeklyDelta`, and `latestPercentileDelta` — but no view currently renders these fields. The feature is dead code in both the backend and the UI.  
- **Expected behavior:** Either implement a nightly game-log ingestion pass (e.g., pybaseball `statcast_search` per player) and add a "Recent Games" section to `PlayerProfileView`, or strip `games`, `GameTrend`, and all delta-related computed properties from the iOS model to reduce binary bloat and schema confusion.  
- **Severity:** Low (dead code / schema drift)  

### Issue BE-4: Backend stores `image_url` that the app ignores

- **Location:** `backend/ingest.py:459`, `StatScout/Models/Player.swift:165`  
- **Observation:** The backend writes `"image_url": "https://img.mlbstatic.com/.../67/current"` but the app computes its own URL via `headshotURL`: `https://midfield.mlbstatic.com/v1/people/\(id)/spots/240`. The backend field is decoded into `Player.imageURL` but never referenced directly in any view; `PlayerHeadshot` uses `headshotURL`.  
- **Expected behavior:** Remove `image_url` from the backend upsert payload, or change `headshotURL` to fall back to the stored `imageURL` before constructing a new URL.  
- **Severity:** Low (wasted bandwidth / schema hygiene)

## Build, Distribution & CI/CD

### Issue CI-1: TestFlight script assumes local Xcode signing session

- **Location:** `scripts/upload-testflight.sh:40-44`  
- **Observation:** The script uses `-allowProvisioningUpdates` and expects the local machine's Xcode → Accounts login to handle App Store Connect authentication. There is no App Store Connect API key (`ASC_API_KEY`) or altool fallback. This is fine for a single-developer setup but is not reproducible on a CI runner or a new Mac.  
- **Expected behavior:** Document the manual-signing prerequisite in `README.md` and add an optional `ASC_API_KEY` path variable for CI-based uploads.  
- **Severity:** Low (bus factor / reproducibility)  

### Issue CI-2: Failure notification creates a new GitHub Issue on every failure

- **Location:** `.github/workflows/nightly-statcast.yml:49-60`  
- **Observation:** The workflow calls `github.rest.issues.create({ title: 'Nightly Statcast ingestion failed', ... })` on failure. A transient MLB API blip could spawn dozens of duplicate issues.  
- **Expected behavior:** Use `github.rest.issues.create` only if an open issue with the same title does not already exist, or append to a single persistent tracking issue.  
- **Severity:** Low (noise)

## Cross-Layer Coherence — What Still Feels Half-Built

1. **The `MetricDirection` ghost type.** The model defines `MetricDirection` (up / flat / down), `TrendGlyph` renders it, and `Player.weeklyDelta` computes it — but no view actually displays trend information. The backend never sends `direction`. Either ingest deltas or delete the scaffolding.

2. **The `GameTrend` ghost type.** Same pattern: full model, decoding logic, `SampleData` fixtures, computed properties on `Player`, and zero UI surface area in production. This is not a rejection issue because users never see a broken promise, but it inflates the binary and schema for a feature that does not exist.

3. **Backend `image_url` vs. iOS `headshotURL`.** Two different MLB CDN endpoints for the same asset. The backend payload carries dead weight.

4. **Dead code in `Components.swift`.** `TrendGlyph`, `FeaturedTile`, and `SectionHeader` are defined and compiled but never instantiated. `Player.swift` defines `TeamRoute` which is never pushed into any `NavigationStack`.

## Recommended Fix Order (Pre-Submission)

1. **Fix the "ⓘ" glyph** (2.1-1) — either make it tappable or delete it. This is the single highest-probability rejection trigger.
2. **Verify `.xcprivacy` resource bundling** (5.1.1-1) — open the generated `.pbxproj` and confirm the file is in a `Resources` build phase.
3. **Update marketing site copy** (2.3-1) — change "Live Leaderboards" to "Nightly Leaderboards" before the review team clicks through.
4. **Add accessibility labels to percentile bars** (2.1-3) — `PercentileBarMini` and `MetricBar` need voice annotations.
5. **Surface `freshnessText` on the dashboard** (2.1-2) — builds user trust and preempts "stale data" complaints.
6. **Move Supabase URL to env variable** (BE-1) — one-line change in `project.yml`.
7. **Clean up dead code** (BE-3, BE-4, cross-layer ghosts) — reduces binary size and schema confusion.
8. **Everything else** is polish that can ship in a follow-up release.
