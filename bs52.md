# StatScout Bug Report - BS52

> Comprehensive bug report from the perspective of an app user. These are issues that affect the user experience when using the StatScout iOS app.

---

## CRITICAL BUGS (App is Broken or Misleading)

### C-1: The "Year Over Year" History Shows Wrong Season Data
**What I see:** When I tap a player and look at "YEAR OVER YEAR", I might see the same season repeated or wrong season labels because the backend doesn't properly filter and the iOS code uses `$0.season ?? 0` which could cause sorting issues.

**What I expect:** Each row should show a distinct historical season (2025, 2024, etc.) with accurate percentile data for that year.

**Why it matters:** Users rely on historical trends to understand player development. Wrong data is worse than no data.

**Root cause:** 
- `DashboardViewModel.load()` at line 114-124 groups by playerId but the sorting uses `($0.season ?? 0)` which treats nil as 0
- The backend upserts multiple seasons but the iOS app might show them incorrectly

---

### C-2: Player Headshots Sometimes Show Wrong Initials
**What I see:** Players with suffixes like "Jr.", "III", "IV" show wrong initials. For example, "Bobby Witt Jr." might show "BW" instead of "BWJ".

**What I expect:** "Bobby Witt Jr." should show "BWJ", "Vladimir Guerrero Jr." should show "VGJ".

**Why it matters:** Initials are used as fallback when images fail to load. Wrong initials look unprofessional.

**Root cause:** `Player.initials` in `Player.swift` line 150-164 has logic for suffixes but the test at line 73-76 shows it should work - need to verify this is actually working in production.

---

### C-3: Two-Way Players May Show Wrong Metric Percentiles
**What I see:** For a two-way player like Ohtani, when I tap on "xwOBA" under Hitting, it might accidentally show the pitching xwOBA percentile instead.

**What I expect:** Tapping a metric under a specific category should always show that category's value.

**Why it matters:** Users make decisions based on these numbers. Showing the wrong percentile is misleading.

**Root cause:** `MetricRankingView.metricPercentile(for:)` at line 73-75 correctly filters by both label AND category, but `PlayerProfileView` at line 163-176 uses a `NavigationLink` that passes the correct category. Need to verify this is working correctly.

---

## HIGH BUGS (Major UX Issues)

### H-1: No Game Trend Data Ever Shown
**What I see:** The "Recent Games" section never appears on player profiles, even though the sample data shows it should.

**What I expect:** A list of recent games with percentile changes, like the sample data shows.

**Why it matters:** Game-to-game trends are a key differentiator from static leaderboards. The app promises this in its description.

**Root cause:** 
- Backend `ingest.py` line 438 hardcodes `"games": []` for every player
- The iOS UI is ready but receives empty games array from API

---

### H-2: "Through [date]" Timestamp Can Be Misleading
**What I see:** The leaderboard shows "Through Apr 28, 2026" but this is computed from `players.map(\.updatedAt).max()` which might be misleading if some players have stale data.

**What I expect:** A clear "Last updated: [date]" that reflects when the data was actually refreshed.

**Why it matters:** Users need to know data freshness for fantasy/wagering decisions.

**Root cause:** `DashboardViewModel.freshnessText` at line 24-30 takes the max of all player update times, which might not reflect the actual ingestion time.

---

### H-3: Team Search May Fail for Some Team Name Variations
**What I see:** Searching for "CHW" or "CWS" might not find all White Sox players if the backend and iOS normalize differently.

**What I expect:** All common team abbreviations and full names should work.

**Why it matters:** Users expect search to "just work" without memorizing exact abbreviations.

**Root cause:** 
- Backend `normalize_team_abbr` in `ingest.py` line 236-294 has an alias map
- iOS `normalizedTeamAbbreviation` in `SavantTokens.swift` line 111-130 has a similar but potentially different map
- The backend now returns "TBD" for unknown teams (line 239), but iOS might not handle this gracefully

---

### H-4: Cache TTL is Too Short for Offline Use
**What I see:** If I open the app without internet, it shows "Can't reach data feed" even though I just used it yesterday.

**What I expect:** The app should show cached data for at least 24-48 hours when offline.

**Why it matters:** Users in areas with poor connectivity (stadiums, transit) need offline access.

**Root cause:** `DiskPlayerCache` in `PlayerCache.swift` line 12 has `maxAge: TimeInterval = 6 * 60 * 60` (6 hours) which is too short.

---

## BACKEND/DATA INGESTION BUGS

### B-1: FanGraphs Data Source Disabled
**What I see:** Standard stats (AVG, HR, RBI, ERA, etc.) are not available for many players.

**What I expect:** Standard traditional stats alongside Statcast metrics.

**Why it matters:** Users want both advanced and traditional stats in one place.

**Root cause:**

- `_fetch_mlb_standard_stats` returns empty dict at line 334

---

### B-2: No Game Log Ingestion
**What I see:** The "Recent Games" section is always empty in production.

**What I expect:** Game-by-game trend data showing how percentiles change.

**Why it matters:** Trend data is a key differentiator from static leaderboards.

**Root cause:** Backend `ingest.py` line 438 hardcodes `"games": []` with no actual game log ingestion.

---

### B-3: Qualifying Threshold Too Low
**What I see:** Leaderboards include players with minimal playing time who have inflated/deflated percentiles due to small sample sizes.

**What I expect:** Only qualified players (e.g., 100 PA for batters, 10 IP for pitchers).

**Why it matters:** Small sample sizes create misleading percentiles.

## PERFORMANCE BUGS

### P-1: Team Counts Recomputed on Every View Update
**What I see:** The Teams tab might be slow to load with many players.

**What I expect:** Instant loading of team counts.

**Why it matters:** Perceived performance affects user satisfaction.

**Root cause:** `TeamsView.swift` line 25 computes `viewModel.teamCounts[abbr]` which is precomputed in `DashboardViewModel`, but the view model updates it on every `updateDerivedState()` call.

---

### P-2: Full Player List Loaded on Every App Launch
**What I see:** App takes time to load on every cold start.

**What I expect:** Show cached data immediately, refresh in background.

**Why it matters:** Users want instant access to their data.

**Root cause:** `DashboardViewModel.load()` at line 97-142 waits for network even when cache is available. It shows cache first (lines 98-101) but sets `isLoading = true` if players is empty after cache.
